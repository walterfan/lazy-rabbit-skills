# PKB Commands — Detailed Reference

<!-- Keep in sync with the Command Summary table in SKILL.md -->

**`PKB_ROOT`**: Directory where all PKB markdown and Sphinx files live. Default `<project-root>/man/` (or `doc/` when that folder already exists for docs). Outside the repo: `<parent>/<project-name>-pkb/man/`. See [SKILL.md](../SKILL.md) for the full root-path contract.

## Update strategy before any command

Use a cost-aware flow before generating or refreshing docs:

1. Run a deterministic freshness check first, such as `scripts/check_pkb_staleness.py`, when available.
2. Apply Level 1 zero-token refreshes first, such as `scripts/update_doc_level1.sh`, when the change is mechanical.
3. Only call the LLM for Level 2 or Level 3 work, and only with the minimal context:
   - relevant diff
   - affected PKB page(s)
   - source/config snippets needed for the update
4. Treat deterministic script outputs as sidecar artifacts under `PKB_ROOT/_generated/`; do not overwrite human-maintained PKB pages directly.

Useful deterministic helpers:

- `scripts/init_pkb.sh` — initialize the PKB skeleton from templates
- `scripts/gen_repo_map.sh` — refresh the repo tree block in repo-map docs
- `scripts/gen_tech_stack.py` — extract dependency and runtime versions from common manifests
- `scripts/gen_index.py` — regenerate `index.md` from the PKB page set
- `scripts/extract_api_signatures.py` — extract Tauri command and REST signatures
- `scripts/check_translation_sync.py` — check whether `zh_CN` gettext catalogs are still in sync with English Markdown
- `scripts/gen_changelog.sh` — draft changelog sections from `git log`
- `scripts/gen_dep_graph.py` — generate a Mermaid dependency graph from static imports

All generated artifacts should:

- start with `Auto-generated by ...`
- include provenance for the source files or git range used
- leave judgment calls as `[NEEDS INPUT: ...]`
- be merged into real PKB pages only after human review or a targeted LLM pass

## Standard numbered page set

New PKBs should default to this page set. Each page heading must include its sequence number — main pages use `# NN. Title`, appendices use `# Appendix-NN: Title`. For bilingual builds, Chinese `.po` translations must mirror the same prefix (e.g. `msgstr "00. 项目概述"`, `msgstr "附录-01: 常见问题"`).

```text
PKB_ROOT/
├── index.md
├── 00-overview.md              # 00. Project Overview
├── 01-quick-start.md           # 01. Quick Start
├── 02-architecture.md          # 02. Architecture
├── 03-tech-stack.md            # 03. Tech Stack
├── 04-repo-map.md              # 04. Repository Map
├── 05-data-and-api.md          # 05. Data and API
├── 06-workflows.md             # 06. Workflows
├── 07-conventions.md           # 07. Conventions
├── 08-build.md                 # 08. Build, Release, and Publish
├── 09-testing.md               # 09. Testing Strategy
├── 10-runbook.md               # 10. Runbook
├── 11-observability.md         # 11. Observability
├── 12-document.md              # 12. Documentation Process
├── ai-guide.md                 # 13. How to Use This Documentation for AI
├── appendix-01-faq.md          # Appendix-01: FAQ
├── appendix-02-glossary.md     # Appendix-02: Glossary
├── diagrams-guide.md           # Appendix-03: Diagram Quick Reference
├── CHANGELOG.md                # Appendix-04: Documentation Changelog
├── adr/
└── changes/
```

If a repo already has an older numbering scheme, preserve it unless the user explicitly asks for migration.

## /PKB-init [project-root] [--sphinx] [--minimal] [--bilingual=zh_CN] [--poetry] [--project-name \<name\>] [--author \<name\>] [--external-pkb [parent-dir]]

Initialize a Project Knowledge Base under **`PKB_ROOT`** using the standard numbered page set.

1. Determine `<project-root>` and verify it is a software project.
2. Resolve **`PKB_ROOT`** inside or outside the repo.
3. Prefer `scripts/init_pkb.sh --pkb-root <PKB_ROOT>` to create the standard page set, `index.md`, `ai-guide.md`, `adr/`, `changes/`, `scripts/`, and `_generated/`.
4. Generate each page from `templates/docs/`, `templates/adr/`, and `templates/changes/`.
5. If `--minimal` is enabled, create only the 3 core pages (`00-overview`, `01-quick-start`, `04-repo-map`) plus `index.md`, `ai-guide.md`, `adr/` skeleton, and essential scripts (`gen_repo_map.sh`, `gen_index.py`).
6. If `--sphinx` is enabled, create `Makefile`, `conf.py`, `pyproject.toml`, `_static/`, `_templates/`, `requirements.txt` fallback, and `make.bat` immediately. The `conf.py` sentinels (`__PROJECT_NAME__`, `__AUTHOR__`, `__COPYRIGHT__`) are replaced with values from `--project-name` and `--author` (default author: Walter Fan).
7. `--poetry` may be passed explicitly, but generated PKBs should be Poetry-first by default and document `poetry install` as the canonical setup flow.
8. If `--bilingual=zh_CN` is enabled, add `locale/zh_CN/LC_MESSAGES/`, gettext settings, and a language switcher.
9. Suggest starting with `/PKB-overview`, `/PKB-quick-start`, and `/PKB-repo-map`.

## /PKB-changelog [project-root]

Generate or update **two separate changelogs**:

| File | Scope | Tracks |
|------|-------|--------|
| `PKB_ROOT/CHANGELOG.md` | Documentation changes | PKB page additions, translations, tooling, review status, structure updates |
| `<project-root>/CHANGELOG.md` | Code/feature changes | Features, bug fixes, refactors, security patches, releases |

### Why two changelogs?

Documentation and code evolve on different cadences and audiences care about different things. A developer checking "what shipped in Q1" wants the project changelog. A doc maintainer checking "when was the architecture page last rewritten" wants the doc changelog.

### Workflow

1. **Mechanical draft first**: Run `make changelog-draft` (or `bash scripts/gen_changelog.sh --repo-root .. --doc-dir man`). This produces a raw changelog in `_generated/08-build.changelog.generated.md` organized by conventional commit type.
2. **Triage entries**: Sort raw entries into the appropriate changelog — code changes go to `<project-root>/CHANGELOG.md`, doc changes go to `PKB_ROOT/CHANGELOG.md`.
3. **Human curation**: Rewrite entries using [Keep a Changelog](https://keepachangelog.com/) format, organized by release/quarter for the project changelog or by date for the doc changelog.
4. **AI synthesis**: AI can help categorize raw commit messages into user-facing summaries (Added, Fixed, Changed, Removed, Security), but a human should verify the impact descriptions.

### Project CHANGELOG structure (at repo root)

```markdown
# Changelog

## 2026-Q2 (April 2026)

### Added
- Feature description (TICKET-xxxxx)

### Fixed
- Bug fix description (TICKET-xxxxx)

## 2026-Q1 (January – March 2026)
...
```

### Doc CHANGELOG structure (in PKB_ROOT)

```markdown
# Documentation Changelog

## 2026-04-15

### Added
- Human review and approval system for PKB-metadata footers
- `scripts/pkb_review_status.py` dashboard

### Fixed
- Language switcher links for bilingual builds

## 2026-04-14
...
```

### Tips

- **Project changelog**: Group by release, quarter, or milestone; reference Jira tickets; call out breaking changes and deployment notes
- **Doc changelog**: Group by date; focus on what was added, restructured, translated, or tooling-wise improved
- Both changelogs should link to each other for cross-reference

## /PKB-verify [project-root]

Verify PKB completeness, quality, and freshness.

1. Check that required docs exist for the chosen PKB layout.
2. Check links, code references, and metadata footers.
3. Run a rule-based freshness pass first when available, preferably `scripts/check_pkb_staleness.py`.
4. Compare each doc's metadata footer against current git history.
5. Classify suggested updates:
   - **Level 1**: mechanical, zero-token
   - **Level 2**: targeted LLM refresh
   - **Level 3**: human-led judgment change
6. Recommend the smallest sensible refresh plan, preferably batched by PR, sprint, or milestone.

## /PKB-build [format] [--lang=en|zh_CN|all]

Build Sphinx documentation from **`PKB_ROOT`**.

Typical flow:

```bash
cd "$PKB_ROOT"
poetry install
make pkb-check-i18n
make html
make serve
```

For bilingual builds, also use:

```bash
make gettext
make intl-update
make pkb-check-i18n
make html-en
make html-zh
```

For deterministic translation drift detection before publish, also run:

```bash
make pkb-check-i18n
# or: python3 scripts/check_translation_sync.py --repo-root . --doc-dir .
```

For bulk translation of empty `.po` entries (e.g. after initial generation or large content updates):

```bash
# 1. Check current coverage
python3 scripts/translate_po.py --stats

# 2. Edit scripts/translate_po.py — add translation dictionaries (msgid -> msgstr)
# 3. Apply translations
python3 scripts/translate_po.py

# 4. Compile .mo and rebuild Chinese HTML
make intl-build
make html-zh
```

The `translate_po.py` script uses `polib` to safely handle multi-line `.po` entries. Never use `sed` or regex to modify `.po` files directly — multi-line continuation strings will be corrupted.

If `PKB_ROOT` is outside the software repo, pass `PKB_REPO_ROOT=/path/to/project` to the Make target.

## /PKB-review-status [--strict] [--json]

Display a read-only dashboard of human review status for all PKB documents.

```bash
cd "$PKB_ROOT"
make pkb-review-status
# or: python3 scripts/pkb_review_status.py --doc-dir .
```

Output shows each doc's `review_status`, `review_score`, `reviewed_by`, and `last_updated`:

```text
PKB Review Status
=================================================================
File                   Status     Score  Reviewer     Updated
-----------------------------------------------------------------
00-overview.md         approved   4/5    walter.fan   2026-04-14
01-quick-start.md      PENDING    0/5                 2026-04-14
02-architecture.md     approved   4/5    walter.fan   2026-04-13
...
-----------------------------------------------------------------
Total: 16 docs | Approved: 8 | Pending: 8 | Avg approved score: 3.5/5
```

Options:

- `--strict`: exit with code 1 if any doc has `review_status` other than `approved` (for CI gates)
- `--json`: output structured JSON instead of a table

### Human review workflow

1. AI generates or updates a doc — stamps `review_status: pending`, `review_score: 0`, `reviewed_by:` (empty)
2. Human opens the doc in their IDE, reads the content, edits if needed
3. Human changes the footer fields directly:
   - `review_status: approved`
   - `review_score: 4` (their quality assessment, 0-5)
   - `reviewed_by: walter.fan`
4. If AI updates the doc later with a substantive change, `review_status` resets to `pending` and the human must re-approve

Exception: if AI only made a trivial Level 1 mechanical fix, it may keep `review_status: approved` but must set `updated_by: ai+trivial`.
