# Voice rules — writing AGENTS.md for both AI tools and human developers

`AGENTS.md` is primarily a manual for coding agents (Codex, Cursor,
Claude Code, Aider, Gemini CLI, …). But it is also the first file a new
human contributor opens when the README is thin or when they want to
know the **rules of the road** for this repo.

Both audiences benefit from the **same prose** — as long as it is
written with one rule:

> **Every command, convention, and danger zone carries a one-clause
> reason.** That clause is what turns a terse imperative ("Run
> `just lint`") into a teaching imperative ("Run `just lint` — keeps
> diffs reviewable and CI green").

This file lists the concrete rules and shows before / after examples.
Use it while drafting AGENTS.md.

---

## Why dual-audience instead of two files?

- A single source of truth removes drift between an "agent doc" and a
  "developer doc". When commands change, only one file moves.
- Coding agents already read agent-style imperatives ("Run X"); a short
  "why" clause does not slow them down and often improves their plan.
- Human contributors already read README-style prose; rationale is what
  they need to internalize the rules.
- Splitting into two files re-introduces the exact problem AGENTS.md
  was created to solve: too many onboarding files in too many places.

The README still exists for the marketing / project-overview / install
story. AGENTS.md is the **rules of the road** for whoever is about to
make a change — agent or human.

---

## The seven rules

### 1. Lead with the action, then add the reason in one clause

The action is what an agent will execute. The reason is what a human
needs to retain the rule.

| Bad (action only) | Good (action + reason) |
|---|---|
| Run `just lint`. | Run `just lint` — keeps diffs reviewable and CI green. |
| Use `pnpm` for installs. | Use `pnpm install` — the lockfile is `pnpm-lock.yaml`; npm/yarn will produce a different tree and break CI. |
| Format with `ruff format`. | Format with `ruff format` before committing — the pre-commit hook re-runs it and will reject unformatted PRs. |

Keep the reason short (4–15 words). If it needs more, link out.

### 2. For every convention, name what it prevents

A convention without a reason is an opinion. A convention with a
reason is a rule. Agents respect rules; humans internalize them.

| Bad | Good |
|---|---|
| Keep public API stable. | Keep public API stable — every external caller depends on these symbols; breaking changes need a major version bump and an ADR. |
| Don't log request bodies. | Don't log request bodies — they may carry tokens or PII; redact at the handler boundary instead. |
| Prefer composition over inheritance. | Prefer composition over inheritance — the codebase has been bitten twice by deep class hierarchies; see `docs/adr/0007-no-inheritance.md`. |

### 3. For every danger zone, name the failure mode

"Don't touch X" is an instruction. "Don't touch X because Y silently
breaks" is a rule both audiences will follow.

| Bad | Good |
|---|---|
| `src/auth/` — danger. | `src/auth/` — token signing and validation. Bugs here leak credentials silently in production logs. Require a design doc + 2 reviewers. |
| `migrations/` — handle with care. | `migrations/` — runs once per environment, no rollback. Add a new migration file; never edit an existing one. |
| Generated code — do not edit. | `src/proto/*.ts` — generated by `just gen-proto` from `proto/*.proto`. Edit the `.proto` and re-run; hand edits are overwritten on next build. |

### 4. Prefer concrete examples over abstract policy

A concrete example is unambiguous. Abstract policy invites
interpretation drift between agents and humans.

| Bad | Good |
|---|---|
| Be careful with logging. | Never log `request.headers`, `cookies`, or anything under `*.token`. Use the `redact()` helper for unknown payloads. |
| Follow company logging standards. | Follow L0–L5 sensitivity tags; full table in `docs/logging.md`. Quick rule: any user-supplied string is L3+ until proven otherwise. |

### 5. Link out for depth, do not inline

AGENTS.md is the index, not the textbook. Inline rationale belongs in
ADRs, RFCs, KB chapters, or design docs. AGENTS.md tells you the rule
exists and where to read more.

| Bad | Good |
|---|---|
| (3 paragraphs explaining why we chose Postgres over MongoDB) | We use Postgres — see [`docs/adr/0003-postgres.md`](docs/adr/0003-postgres.md) for the trade-off analysis. |
| (full 30-line redaction policy inlined) | Redact secrets in logs — see [`docs/logging.md#redaction`](docs/logging.md#redaction). |

### 6. Keep imperative mood; rationale is a clause, not a paragraph

Both audiences scan first, read second. A long paragraph buries the
imperative.

| Bad | Good |
|---|---|
| It is generally a good idea, when working in this codebase, to make sure that you run the test suite frequently, because we have observed that integration tests can break in subtle ways when the database schema changes. | Run `just test` after any schema change — integration tests catch silent migration bugs. |

### 7. Keep the common sections visible

The default AGENTS.md shape is intentionally small: Context Map,
Commands, Harness Rules, Project Rules, AI Tooling, and Keeping Current.
A new section must earn its place by preventing a recurring mistake that
cannot be handled by linking to supporting docs.

```markdown
## Keeping Current

Update this file when commands, layout, guardrails, agent tools, or
linked docs move. If `man/` changes, update links here instead of
copying its content.
```

## Section-by-section voice guide

| Section | What humans need | What agents need | Combine by … |
|---|---|---|---|
| Title + purpose | What is this project? | One-sentence ground truth | One sentence + one-line "what it is not" if scope is easy to misread |
| Context Map | "Where does my change go?" | "Which path matches the task?" | 5-8 bullets with directory/module purpose plus links to README, PKB, and design docs |
| Commands | "How do I install / build / test / lint?" | "Which command satisfies the task?" | Code block with comment per command |
| Harness Rules | "How should agents behave here?" | "Which operating rule applies?" | Short rules for honesty, ambiguity, planning, simplicity, surgical edits, and verification |
| Project Rules | "What repo-specific rules matter?" | "Which local convention or danger zone applies?" | Concrete conventions, do-not rules, review expectations, and security constraints |
| AI Tooling | "Which assistant files matter?" | "Which context file should I read?" | Primary tools plus optional symlink/config notes |
| Keeping Current | "When do I update this file?" | "What stale context should I repair?" | Trigger list plus a project-learning loop |

Optional details such as full PKB indexes, change workflows, long
agent-client setup instructions, or troubleshooting tables should
usually live in linked docs.
Add them to root AGENTS.md only when the user asks or the repo has a
recurring failure mode that must be visible before the first edit.

## Anti-patterns specific to dual-audience writing

- **Forking into "for humans" / "for agents" subsections.** Drift is
  guaranteed within months. Use one prose stream with reasons.
- **Adding a "How to onboard" section that duplicates the README.** The
  README is the onboarding doc. AGENTS.md says *what changes
  here* — link to the README for project setup story.
- **Inlining tutorials.** Tutorials are KB content. AGENTS.md links
  to them.
- **Adding emoji or marketing voice to soften imperatives.** Both
  audiences prefer terse + reasoned over soft + vague.
- **Long block quotes of policy.** Replace with a one-line link to the
  policy doc.
- **"You should…" / "We recommend…".** Use imperative ("Run X",
  "Avoid X") + reason. Recommendation language tells neither audience
  whether the rule is enforced.

---

## Quick checklist (use during Phase B step 5)

- [ ] Every command has a one-clause reason.
- [ ] Every guardrail bullet names what it prevents.
- [ ] Every danger zone, if present, names the failure mode and the escalation path.
- [ ] Every long rationale is replaced by a link to ADR / KB / RFC.
- [ ] No "for humans" / "for agents" forks.
- [ ] Root AGENTS.md is under 100 lines; nested files are under 60 lines.
- [ ] Imperative mood preserved; rationale is a clause, not a paragraph.
